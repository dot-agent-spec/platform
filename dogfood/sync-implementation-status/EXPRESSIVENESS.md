> ⚠️ Point-in-time DSL usability snapshot (2026-06-23). NOT spec truth — may be obsolete. Source of truth: dsl/ + packages/.

# DSL Expressiveness Report — Sync Implementation Status

> **Context:** `sync-status.description` + `sync-status.behavior` model the `/sync-implementation-status`
> skill as a dot-agent state machine. This document records what the DSL expressed cleanly and every
> place where the language had to be bent or could not express the intent at all.
> Each gap is a candidate RFC. Gaps numbered S1–S5 are new; cross-references to rfc-author G1–G7 are noted.

> **Shape verdict:** this flow is fundamentally a **code-analysis pipeline**, not a conversation.
> The FSM routing expressed cleanly; the data transformation (extract, cross-reference, diff) is
> entirely outside the DSL — scripts carry all the semantic weight. The DSL's contribution is
> routing + one human checkpoint. That is a narrow but valid use.

---

## What expressed cleanly

- **Linear pipeline with per-stage error exits** — the chain of six setup states
  (`initialize → extract_features → verify_parser_dsl → verify_compiler → verify_kernel_dsl → verify_sdk`)
  maps directly to the DSL state model. Each stage uses `run script … on failure → error`, which is
  exactly the right construct: no ceremony, one line per step.

- **Single interactive checkpoint** — only `review_drift` requires human input; all verification
  stages are automated setup states. The DSL naturally separates setup states (no `interact`) from
  oriented states (with `interact`). The language boundary here mirrors the workflow boundary.

- **Error recovery with full restart** — `state error` with `on intent "retry" → initialize` cleanly
  expresses "retry from scratch" without any special syntax. Self-contained.

- **Re-run workflow** — `state done` with `on intent "run again" → initialize` restarts the pipeline
  without any mechanism other than a transition. The FSM loop is free.

- **Agent public contract** — `.description` cleanly separates the agent's capabilities and I/O
  contract from its behavior. `SearchAction` and `WriteAction` capabilities accurately reflect the
  two phases (read code → write doc) without over-specifying internals.

## Gaps & limitations

### S1 — `parallel` grammar does not support multi-line handler blocks (bug)

**Problem:** The `parallel_stmt` grammar is:
```
parallel_stmt: seq('parallel', handler_block, optional(success_stmt), failure_stmt)
```
where `handler_block` is `prec.right(repeat1(choice(memory_stmt, run_stmt, ...)))`. Unlike
`block`, `handler_block` has **no newline handling** — no `optional($._end_stmt)` between
statements. Since newlines are not in the grammar's `extras` (only spaces/tabs are), each
statement in the handler_block must appear on the same line.

**Reproduction:** placing `run` statements on separate lines inside `parallel` produces E004:
```
state verify_layers
  parallel
    run script "scripts/sync-status/verify-parser-dsl.js"   ← E004 here
    run script "scripts/sync-status/verify-compiler.js"
  on success
    transition to review_drift                              ← E004 here
  on failure
    transition to error                                     ← E004 here
```

The linter reports `E004 "Syntax error near 'parallel'"` and `E006 "Parse error at line N, column 20"`.

**Workaround used:** replaced the single `verify_layers` state (parallel) with four sequential
states (`verify_parser_dsl → verify_compiler → verify_kernel_dsl → verify_sdk`). This is
semantically correct — the WASM kernel executes `parallel` sequentially anyway — but the
conceptual shape of "four independent checks" is lost in the DSL.

**RFC candidate:** fix `parallel_stmt` to use `$.block` instead of `$.handler_block`, or add
`optional($._end_stmt)` between statements in `handler_block`. This would allow each `run` in
a `parallel` to appear on its own line, matching the expected usage.

### S2 — No mechanism to bind script output to `goal` or `guide` text

**Problem:** The `guide` in `review_drift` is a static string that tells the LLM to "present the
drift report". The actual drift report is generated by the verification scripts. There is no
mechanism in the DSL to capture script output and inject it into a `goal` or `guide` statement.

The LLM must infer that "the drift report" is available from session memory or some side-channel
(e.g., a file written by the scripts). This is implicit coupling between the script layer and the
LLM instruction layer.

**Workaround used:** the static guide instructs the LLM to "present the complete drift report"
without specifying where to find it. The scripts would need to write to session memory via the
Runtime API, and the LLM would need to read from there implicitly.

**RFC candidate:** allow `goal` and `guide` to reference session variables, producing dynamic text:
```
guide session.drift_report
```
or an interpolation syntax:
```
guide "Drift detected: {{session.drift_summary}}. Ask the user to approve or cancel."
```
This pairs with rfc-author G1 (input extraction) and G4 (run parameter interpolation) — all three
are facets of the same missing feature: data flow between the script layer and the DSL layer.

---

### S3 — No `on success` handler on individual `run_stmt`

**Problem:** The natural conditional after verification is: "if no drift detected, go to `done`;
if drift detected, go to `review_drift`". A script could signal this via exit code or memory,
but the DSL only supports `on failure` for individual `run` statements — there is no `on success`
for `run_stmt`.

The grammar comment says "on success block — success handler (for `parallel` only, optional)".
But even for `parallel`, the `success_stmt` is broken by S1. For individual `run_stmt`,
`on success` is absent entirely.

**Workaround used:** always transition to `review_drift` after `verify_sdk`. The `review_drift`
state presents the report and lets the user approve even when there is no drift — a minor UX gap
but a correct fallback.

**RFC candidate:** add optional `on success` to `run_stmt`, symmetric with `on failure`:
```
run script "scripts/sync-status/verify-sdk.js"
  on success
    transition to review_drift
  on failure
    transition to error
```
(Or, if `on success` is absent, execution falls through to the next statement in the block — the
current behavior for `on failure`-less runs.)

---

### S4 — No data passing between sequential `run` stages

**Problem:** the feature list extracted in `extract_features` (from `node-types.json`) must be
consumed by `verify_parser_dsl.js`, `verify_compiler.js`, etc. The DSL provides no mechanism to
pass structured data from one `run` call to the next. Each script sees only what it can read
from disk or the Runtime memory API — the session state is not addressable from `.behavior`.

**Workaround used:** scripts would write results to a temp file or session memory via the
Runtime API, and subsequent scripts would read from there. This works operationally but the
coordination is invisible to the DSL — the `.behavior` file cannot document or enforce it.

**RFC candidate:** same as rfc-author G4 (parameterized `run` with memory interpolation). Extends
it to also cover **output capture**: a way to store the result of a `run` into session memory:
```
run script "scripts/sync-status/extract-features.js"
  capture session.features
  on failure
    transition to error
```

---

### S5 — No way to signal a clean session end (confirmed from rfc-author G5)

**Problem:** `state done` loops on `on offtopic` instead of ending the session. `transition to ended`
would be semantically correct but triggers E005 (undefined state) because `ended` is a native
Runtime state not declared in the behavior file.

**Workaround used:** same as rfc-author — `done` loops to itself on `on offtopic`. Semantically
incorrect (the session never ends), but the only available option.

**RFC candidate:** same as rfc-author G5. Not re-documented here; see
`dogfood/rfc-author/EXPRESSIVENESS.md §G5`.

## Out-of-scope verdict

The core work of this skill — reading source files, cross-referencing grammar nodes with AST
variants, applying the discrepancy map, classifying cells — is pure code analysis. The DSL has
no constructs for iterating over a list of nodes, comparing sets, or classifying items into
categories. All of this must live in scripts.

This is not a gap in the DSL; it is the **intended boundary**. `.behavior` expresses *routing*
and *human interaction*; scripts express *computation*. The sync-status flow sits almost entirely
on the script side of that boundary, with the DSL providing only routing + one checkpoint.

The one place where the boundary is genuinely unclear: S2 (dynamic content in `guide`). The
LLM needs to know what is in the drift report to present it — that is a DSL responsibility
(instructing the LLM) but the content is script-generated. This seam is underspecified.

## Parser & linter error-message quality

**Context:** the first attempt included `parallel` with multi-line `run` statements. The linter
returned:
```
E004: Syntax error near 'parallel'.
```

This message is correct but unhelpful. The root cause — that `handler_block` (used inside
`parallel`, `if`, intent blocks) does **not** consume `$._newline` between statements, unlike
`block` — required manually inspecting the grammar to understand.

**Errors that would have been immediately actionable:**

1. **Contextual handler_block restriction** — E004 could mention the constraint:
   ```
   E004: handler_block at line 24, column 20 does not accept newlines between statements.
   Each statement must appear on one logical line. This is used in: parallel, if, on intent,
   on offtopic handlers.
   
   Your code:
     parallel
       run script "..." ← newline before this
   
   Did you mean sequential states? Example:
     state stage1
       run script "a.js" on failure → error
       transition to stage2
   ```

2. **Pattern detection + suggestion** — the linter sees `parallel` followed by multiple `run`
   statements on separate lines and could offer:
   ```
   W013: parallel with multi-statement handler_block is a common pattern that doesn't parse.
   Consider using sequential setup states instead for better readability.
   Suggested refactor: replace parallel { run a; run b; run c } with:
     state stage_a { run a on failure → error; transition to stage_b }
     state stage_b { run b on failure → error; transition to stage_c }
   ```

3. **AST mapping failures with enum context** — when a parser (Rust AST mapper) fails, the
   error E006 could name the enum variant that was attempted:
   ```
   E006: Parse error at line 22, column 5. Failed to deserialize statement.
   Context: inside on intent "topic provided" handler.
   The IntentBody enum does not include Statement::Set.
   
   Valid statements in intent handlers: run, transition, set, apply/remove css, if, parallel.
   
   Your code attempted: set session.flag = true (inside intent handler)
   This is a known mapper bug (see rfc-author/EXPRESSIVENESS.md G2).
   ```

**Practical recommendation:** emit an RFC for linter improvements targeting these three areas.
The gaps are not in the language grammar — the parsing is correct. They are in the *diagnostic
output* that guides the author toward fix or workaround. A clearer message would have reduced
iteration time by ~50% during dogfood construction (one grep of the grammar would have been
avoided).

**Implementation strategy:** store metadata in the grammar (via comments or a separate YAML
schema) for each rule:
- `contextual_rules`: list of contexts where this rule appears and constraints that apply
- `common_mistakes`: patterns that parse but are likely unintended
- `workarounds`: suggested refactors when a pattern is unsupported

The linter would consult this metadata when reporting E004/E006, producing richer diagnostics
without changing the grammar itself.
