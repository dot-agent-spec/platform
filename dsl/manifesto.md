# .flow — Manifesto

## The space .flow occupies

```
Prompt  →  .flow  →  .run  →  Runtime
```

Each layer serves a purpose:

- **Prompt** — highly flexible, probabilistic. The LLM reasons, interprets, and generates. Poorly suited to enforcing routing logic or deterministic tool calls.
- **.flow** — structured, readable, deterministic enough. Coordinates knowledge, memory, actions, and state without exposing the full complexity of the runtime.
- **.run** — maximum expressiveness and performance. Compiled binary for logic that exceeds `.flow`'s cognitive scope.
- **Runtime** — the execution layer. Interprets `.flow`, manages memory, routes to models, and delegates to `.run`.

`.flow` does not try to replace any of these layers. It fills the gap between prompt and runtime for the common case.

---

## The practical motivation

> "I can use LangGraph. I don't want to have to use LangGraph to solve something simple."

This is the origin of `.flow`.

There is a wide range of agent workflows that are too structured for a prompt but not complex enough to justify a full graph framework. In that space, the developer faces a choice between under-engineering (adding control flow to a prompt) or over-engineering (building a LangGraph pipeline for a three-state flow).

`.flow` is that middle path.

---

## What .flow coordinates

A `.flow` file describes:

- **Knowledge** — what the agent should know at each point (`guide`, `teach`)
- **Memory** — what it remembers and for how long (`context`, `session`, `project`, `user`)
- **Actions** — what it executes deterministically (`run tool`, `run script`, `run subagent`)
- **State** — where execution is in the flow (`state`, `transition`)
- **Events** — what it reacts to globally (`on event`, `on intent`)

It does not expose the runtime internals behind these concepts. The user declares intent; the compiler resolves implementation.

---

## The frontier between .flow and .run

The criterion is not the number of lines or states.

The criterion is **cognitive density**.

A `.flow` file should remain scannable in under 30 seconds. If a flow requires simultaneous state tracking, reducers, complex aggregations, compensations, or advanced synchronization — that logic belongs in `.run`.

Some signals that a flow has crossed the frontier:

- You need to loop over a collection
- You need to aggregate and transform results before acting on them
- You need transactional rollback across multiple operations
- The flow requires arithmetic beyond simple comparisons

When in doubt: keep `.flow` readable. Move complexity down.

---

## Small vocabulary as a product decision

The reserved word count is a product metric, not just a syntactic choice.

A smaller vocabulary means:

- Faster to learn
- Easier to document
- More predictable when generated by AI
- Less surface area for misuse

Every proposed addition to the language must answer the question:

> "Does this reduce cognitive load, or does it only add power?"

Power alone is not justification. Power belongs in `.run`.

---

## What .flow is not

**.flow is not a simplified programming language.**
No loops, no recursion, no arithmetic. These belong in scripts or `.run`.

**.flow is not YAML with fewer symbols.**
YAML is a data format. `.flow` is a behavioral language. The distinction matters for readability and for how AI generates it.

**.flow is not natural language.**
The temptation to drift toward free-form text (e.g., `remember that I prefer Portuguese`) must be resisted. Natural language reduces parseability and predictability. The language uses human concepts with structured syntax — not prose.

**.flow is not a competitor to LangGraph or XState.**
Those frameworks are compilation targets. `.flow` abstracts over them.

---

## Definition

> `.flow` is a coordination language for AI agents.

It coordinates knowledge, memory, actions, and state — without exposing the implementation complexity of the runtime beneath it.
