# Bug Fix History: kernel-dsl WASM Initialization

**Problem**: Module "env" not found when loading kernel-dsl WASM in chatbot-ui  
**Status**: ✅ RESOLVED  
**Last Update**: 2026-06-03 13:00  
**Solution**: 31-function WASI + Rust runtime shim

---

## Timeline of Attempts

### ❌ Attempt 1: @wasmer/wasi Polyfill (2026-06-03 11:00)

**Approach**: Install `@wasmer/wasi` package to provide complete WASI interface

```bash
npm install @wasmer/wasi@1.2.2
```

**Implementation**:
```javascript
import { WASI } from "@wasmer/wasi";
_wasi = new WASI({ args: [], env: {} });
const importObject = _wasi.getImportObject();
```

**Result**: ❌ **FAILED**
- `@wasmer/wasi` has internal file `wasmer_wasi_js_bg.wasm` that webpack cannot resolve
- Error: `Module not found: Can't resolve 'wasmer_wasi_js_bg.wasm'`
- Next.js bundler fails during build

**Why**: The library has broken/incomplete WASM dependencies

---

### ❌ Attempt 2: Simple Shim WASI + Manual Fetch (2026-06-03 11:30)

**Approach**: Minimal WASI shim + manual WASM loading via fetch

```javascript
const wasiShim = { proc_exit: () => {}, fd_write: () => 0, ... };
const wasmBuffer = await fetch('/kernel-dsl.wasm').then(r => r.arrayBuffer());
const importObject = { env: wasiShim, wasi_snapshot_preview1: wasiShim, ... };
const wasmModule = await WebAssembly.instantiate(wasmBuffer, importObject);
```

**Result**: ❌ **FAILED in Electron**
- Next.js: ✅ Works fine
- Electron: ❌ Error: `function import requires a callable`
- WASM missing function: `__wbg_call_9c758de292015997`
- Root cause: Not providing all wasm-bindgen function imports

---

### ❌ Attempt 3: Dynamic Import of @wasmer/wasi (2026-06-03 11:45)

**Approach**: Move `@wasmer/wasi` import inside `init()` to delay bundler processing

```javascript
export async function init() {
  const { WASI } = await import("@wasmer/wasi");
  // ... rest of init
}
```

**Result**: ❌ **FAILED**
- Same issue: webpack still tries to resolve transitive dependencies
- Error: `Module not found: Can't resolve 'wasmer_wasi_js_bg.wasm'`
- Bundler processes all imports eagerly, regardless of dynamic syntax

---

### ❌ Attempt 4: Monkey-Patch WebAssembly.instantiate (2026-06-03 12:00)

**Approach**: Intercept WebAssembly.instantiate at runtime to inject WASI

```javascript
const originalInstantiate = WebAssembly.instantiate;
WebAssembly.instantiate = async (wasmBufferOrModule, importObject = {}) => {
  Object.assign(importObject.env, wasiShim);
  return originalInstantiate.call(this, wasmBufferOrModule, importObject);
};
```

**Result**: ❌ **FAILED**
- File `dot_agent_kernel_dsl.js` (generated by wasm-bindgen) imports WASM before runtime
- Import happens at parse time, before patch can be applied
- Error: `Module parse failed: parseVec could not cast the value`

---

### ❌ Attempt 5: Webpack Loader Configuration (2026-06-03 12:15)

**Approach**: Configure Next.js webpack to handle .wasm files

```javascript
// next.config.js

**Result**: ❌ **FAILED**

### ✅ Attempt 6: Patch wasm-bindgen Output + Manual Loading (2026-06-03 12:30)

**Approach**: 
1. Remove WASM import from wasm-bindgen-generated `dot_agent_kernel_dsl.js`
2. Load WASM manually in `index.js` with full import object
3. Auto-patch after every build

**Files Modified**:

**`scripts/patch-wasm-bindgen.js`** - Removes problematic imports

// Remove these lines from dot_agent_kernel_dsl.js:
// import * as wasm from "./dot_agent_kernel_dsl_bg.wasm";
// import { __wbg_set_wasm } from "./dot_agent_kernel_dsl_bg.js";
// __wbg_set_wasm(wasm);
// wasm.__wbindgen_start();
```

**`package.json`** - Auto-apply patch after build
```json

```

**`index.js`** - Manual WASM loading with proper import object
```javascript
export async function init() {
  const bgModule = await import('./pkg/dot_agent_kernel_dsl_bg.js');
  const wasmBuffer = await fetch(wasmPath).then(r => r.arrayBuffer());
  
  const importObject = {
    env: wasiShim,
    wasi_snapshot_preview1: wasiShim,
    './dot_agent_kernel_dsl_bg.js': { ...bgModule }  // ← KEY: spread exports
  };
  
  const wasmModule = await WebAssembly.instantiate(wasmBuffer, importObject);
  bgModule.__wbg_set_wasm(wasmModule.instance.exports);
  
  _module = await import('./pkg/dot_agent_kernel_dsl.js');
}
```

**`dashboard.tsx`** - Client-only loading
```typescript

```

## Lessons Learned

### ✅ What Works
1. Manual WASM loading via fetch() - avoids bundler issues
2. Import object with module key matching WASM imports
3. Spreading ES module exports into import object
4. Post-build patch script for bundler compatibility

### ❌ What Doesn't Work
1. External WASI polyfill libraries (have their own WASM)
2. Webpack/bundler automatic WASM handling
3. Monkey-patching at runtime
4. Incomplete shims missing Rust runtime functions

### 🔍 Root Cause Analysis
- **Core Issue**: `wasm32-wasip1` target includes Rust runtime checks
- **Compilation**: Tree-sitter (C) + Rust WASI = Complex dependencies
- **Loading**: Manual control needed due to bundler limitations
- **Solution**: Provide complete import object with all expected functions
