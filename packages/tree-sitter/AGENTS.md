# tree-sitter ‚Äî Agent Guidelines

AI collaboration guide for maintaining and evolving the Tree-sitter grammars for the .agent DSL.

---

## What this package is

`tree-sitter` provides the **canonical grammar source** for the two file formats of the .agent DSL:

| Grammar | Covers | Located in |
|---------|--------|------------|
| Root `grammar.js` | `.description` manifests and `.type` declarations | `grammar.js`, `src/` |
| `behavior/grammar.js` | `.behavior` behavior files | `behavior/grammar.js`, `behavior/src/` |

Both grammars are authored here and compiled by Tree-sitter CLI into C parsers that power IDE tooling (syntax highlighting, LSP diagnostics, go-to-definition) across VS Code, Zed, Neovim, and Helix. The language specification lives in [`docs/`](docs/); this package is the executable implementation of that spec.

---

## Repository Organization & Versioning

To ensure scalability and consistency between the language specification and its implementation, we follow a dual-versioning strategy.

### 1. Dual-Versioning Strategy
- **Language Specification:** Versioned independently as a standard (e.g., `1.0.0-draft`). Changes to keywords, syntax rules, or semantic behavior bump this version.
- **Tree-sitter Parser:** Follows Semantic Versioning (SemVer) in `package.json`.
  - **MAJOR:** Breaking changes to the parser API or removal of supported language features.
  - **MINOR:** Support for new language features or significant parser improvements.
  - **PATCH:** Bug fixes in token recognition or internal parser logic.

**Git Tagging Convention:**
- Use `spec-vX.Y` for language specification milestones.
- Use `vX.Y.Z` for parser releases (npm releases).

### 2. Documentation Structure (Di√°taxis)
Documentation is organized in the `/docs` folder:
- `/docs/reference`: Information-oriented (Syntax, Keywords, Types).
- `/docs/concepts`: Understanding-oriented (Architecture, Design Principles).
- `/docs/tutorials`: Learning-oriented (Step-by-step guides).
- `/docs/guides`: Task-oriented (How-to guides for specific use cases).
- `/examples`: Executable code snippets (Canonical examples of the DSL).

### 3. Agent Mandates for Updates
When making changes to the codebase, agents MUST follow these rules:
- **Sync Rule:** Any change to `grammar.js` or `behavior/grammar.js` MUST be accompanied by a corresponding update in the `/docs/reference` documentation.
- **Verification Rule:** After any change, run `npm run build` and `npx tree-sitter test` (and `npm run test-behavior`) to ensure the implementation is still valid.
- **Changelog Rule:** Record changes in `CHANGELOG.md` (create it if missing), clearly distinguishing between **Language** changes and **Parser/Tooling** changes.

## File ownership ‚Äî what to edit vs. what to never touch

### Edit freely
| File | Purpose |
|------|---------|
| `grammar.js` | `.description`/`.type` grammar rules |
| `src/scanner.c` | Manual INDENT/DEDENT/NEWLINE scanner for `.description`/`.type` |
| `queries/highlights.scm` | Syntax highlight queries for `.description`/`.type` |
| `test/corpus/types.txt` | Corpus test cases for `.description`/`.type` |
| `behavior/grammar.js` | `.behavior` grammar rules |
| `behavior/src/scanner.c` | Manual INDENT/DEDENT/NEWLINE scanner for `.behavior` |
| `behavior/queries/highlights.scm` | Syntax highlight queries for `.behavior` |
| `behavior/test/corpus/basic.txt` | Corpus test cases for `.behavior` |
| `tree-sitter.json` | Parser scope/file-type declarations |
| `package.json` | npm scripts and dev dependencies |

### Never edit ‚Äî generated by `tree-sitter generate`
| File | Reason |
|------|--------|
| `src/parser.c` | Generated C parser ‚Äî overwritten on every `generate` |
| `src/grammar.json` | Generated grammar IR |
| `src/node-types.json` | Generated node type manifest |
| `behavior/src/parser.c` | Same, for the behavior grammar |
| `behavior/src/grammar.json` | Same |
| `behavior/src/node-types.json` | Same |

### Generated by `npm run build` ‚Äî do not commit
| File | Reason |
|------|--------|
| `dist/tree-sitter-agent.wasm` | Compiled WASM ‚Äî regenerated on `npm run build` |
| `dist/tree-sitter-behavior.wasm` | Same, for the behavior grammar |
| `*.dylib` | Native macOS parser binary ‚Äî local only, not versioned |

The `dist/` folder is excluded from git but **included in the npm package** via the `"files"` field in `package.json`. WASM build requires [Emscripten](https://emscripten.org/) (`emcc` on `$PATH`) and runs automatically on `npm publish` via `prepublishOnly`.

### Never edit ‚Äî from tree-sitter upstream (MIT, see NOTICE)
| Files |
|-------|
| `src/tree_sitter/alloc.h`, `array.h`, `parser.h` |
| `behavior/src/tree_sitter/alloc.h`, `array.h`, `parser.h` |

## Development workflow

```
1. Edit grammar.js (or behavior/grammar.js)
       ‚Üì
2. npx tree-sitter generate          # recompiles src/parser.c
       ‚Üì
3. npx tree-sitter parse <file> --quiet   # manual smoke test
       ‚Üì
4. Update test/corpus/ if needed
       ‚Üì
5. npx tree-sitter test              # confirm all corpus cases pass
```

For the behavior grammar, run the same steps inside `behavior/` using `npx tree-sitter generate` and `npx tree-sitter test` from that directory (or use the npm scripts: `npm run generate-behavior`, `npm run test-behavior`).

**After editing `grammar.js`**, always regenerate before committing ‚Äî stale `src/parser.c` causes silent parse failures in editors.

## Grammar architecture notes

Both grammars share the same indentation strategy: INDENT, DEDENT, and NEWLINE tokens are emitted by an **external scanner** (`src/scanner.c` / `behavior/src/scanner.c`) because Tree-sitter has no native indent support. The scanner uses a stack to track indent levels and emits the correct synthetic token per line boundary.

The sep-by pattern used in blocks:
```js
seq(item, repeat(seq($._newline, item)), $._dedent)
```
is intentional ‚Äî items must **not** consume their own trailing newline because the scanner emits `_dedent` (not `_newline`) after the last item in an indented block.

### "on" disambiguation in `.behavior`

```
on event "..."        ‚Üí trigger_decl      (top-level, uses generic block)
on intent "..."       ‚Üí intent_handler    (inside oriented_state_body)
on offtopic           ‚Üí offtopic_handler  (inside oriented_state_body)
on failure            ‚Üí failure_stmt      (inside run/apply/remove, uses block)
on success            ‚Üí success_stmt      (inside parallel only, uses block)
```

### State body types

Two distinct state structures enforce correctness:

**`oriented_state_body`** ‚Äî LLM-interactive state:
- Order: `goal` ‚Üí `guide?` ‚Üí `teach*` ‚Üí `interact` ‚Üí `handler+` (repeat1) ‚Üí `offtopic_handler`
- `goal` is mandatory; `guide` is optional; `interact` is mandatory
- Handlers (‚â•1 intent + 1 offtopic) are mandatory ‚Äî guarantees FSM never deadlocks
- `goal`/`guide` ‚Üí injected into LLM message context (semantic orientation)
- `teach` ‚Üí fills LLM reusable cache (knowledge base), not message context

**Setup state** ‚Äî orchestration-only state:
- `state_body` dispatches to setup via `repeat1(block)` followed by optional `oriented_state_body`
- Actions: `run`, `set`, `apply`, `remove`, `transition`, `if`, `after`, `parallel`
- No orientation keywords: no `goal`, `guide`, `teach`
- No interaction: no `interact`, no handlers

**`handler_block`** ‚Äî restricted block used inside `intent_handler`, `offtopic_handler`, `failure_stmt`, `success_stmt`:
- Allowed: `run`, `set`, `apply`, `remove`, `transition`, `if`
- Forbidden: `goal`, `guide`, `teach`, `interact`, `after`, `parallel`, nested handlers

Note: Generic `block` is used in `trigger_decl` (top-level `on event`), where all statement types are permitted.

### State transitions

State transitions use `transition to <state>`:

```
state validacao
  transition to aprovado
```

The inline form of `on intent` uses the same keyword pair:

```
on intent "confirmed" transition to next_state
```

## Adding new syntax

1. Add the rule to `grammar.js` (or `behavior/grammar.js`)
2. Add it to the appropriate parent rule's `choice(...)` list
3. Regenerate: `npx tree-sitter generate`
4. Add a corpus test case in `test/corpus/` illustrating the new syntax
5. Update `queries/highlights.scm` with appropriate capture names
6. If this represents a spec change, update [`language.md`](language.md)
