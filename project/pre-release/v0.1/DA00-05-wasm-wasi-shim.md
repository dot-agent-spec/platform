## 1. Summary

After the monorepo flatten (DA00-05, 2026-06-26), a fresh rebuild of `kernel-dsl` on Rust 1.95 exposed a build-chain breakage: `wasi-stub 0.3.0` panics while processing the `wasm32-wasip1` WASM binary produced by the new compiler. A secondary issue ŌĆö the root workspace `npm run build` failing on `language-server` ŌĆö was fixed in the same session.

**Fix applied:** the `wasi-stub` step was removed from `build-wasm.sh` and replaced with a JavaScript WASI shim embedded in `index.js` and `index.browser.js`. This is the canonical fix because `wasi-stub 0.3.0` is the latest version and the bug is unfixed upstream.

## 3. Root cause ŌĆö kernel-dsl

### What `wasi-stub` does

`wasi-stub` processes the `_bg.wasm` output of `wasm-bindgen`. It:

1. Disassembles the WASM binary to WAT text (using `wasm-tools`/wasmparser internally)
2. Finds all imports from the `wasi_snapshot_preview1` module
3. Replaces each import with an inline stub function that returns a fixed value
4. Reassembles back to binary

The purpose is to produce a WASM binary with zero external WASI dependencies, so `index.js` can instantiate it with a minimal `importObject` containing only `env` (UBSan handlers) and `./dot_agent_kernel_dsl_bg.js` (wasm-bindgen internals).

### The Rust 1.95 change

Starting around Rust 1.82ŌĆō1.95, the `wasm32-wasip1` standard library switched from raw FFI WASI imports to bindings generated by `wit-bindgen`. These generated bindings attach a **crate instance disambiguation hash** to the WAT text representation of each import. For example, the `random_get` import (used by the `getrandom` crate for `HashMap` seeding) appears in WAT as:

```wat
$wasi[6eddfa6c868d885d]::lib_generated::wasi_snapshot_preview1::random_get
```

The `[6eddfa6c868d885d]` is a stable fingerprint of the crate version/interface (from `wit-bindgen`). It is present only in the WAT **text** disassembly ŌĆö the binary WASM module still uses module=`wasi_snapshot_preview1`, function=`random_get` at the byte level.

### Why wasi-stub panics

`wasi-stub 0.3.0` (current latest, repo: [astrale-sharp/wasm-minimal-protocol](https://github.com/astrale-sharp/wasm-minimal-protocol/tree/master/crates/wasi-stub)) uses a WAT text parser that does not support square brackets `[...]` inside identifiers. This is an upstream bug; `[hash]` syntax is a `wasm-tools` extension for component model type annotations on core WASM modules. The panic fires when wasi-stub tries to locate/replace the function name and calls `.unwrap()` on the parse result.

`wasi-stub` processes each WASI import in sequence. It successfully started processing `random_get` (printed the "Stubbing function" line) then panicked during the identifier parse.

### Why there is no Rust flag to suppress the hash

The hash is emitted by `wit-bindgen` code generated inside the Rust standard library for `wasm32-wasip1`. No public `RUSTFLAGS` or `.cargo/config.toml` flag exposes control over this naming.

The `getrandom` crate does expose `--cfg getrandom_backend="custom"` to replace the `random_get` import with a custom implementation, but:
- This only covers `random_get` ŌĆö all other WASI imports (`fd_write`, `clock_time_get`, `environ_sizes_get`, `proc_exit`, etc.) would remain
- Requires modifying Rust source to register the custom implementation
- Does not fix the upstream parsing bug in `wasi-stub`

## 4. Root cause ŌĆö language-server

`packages/language-server` is a pure Node.js runtime (no TypeScript compilation). Its `package.json` has never had a `build` script. The root workspace `package.json` ran `npm run build --workspaces`, which fails on any workspace without a `build` script. This predates the monorepo but was masked when the language-server had its own `.git` (builds were never run from the workspace root).

## 6. Fix

### 6.1 `scripts/build-wasm.sh` ŌĆö remove wasi-stub

The entire wasi-stub block (install check + invocation + error check) was removed. The build now ends after `wasm-bindgen` + `patch-wasm-bindgen.js`.

```diff
-echo -e "${GREEN}[INFO] Stubbing WASI imports (browser-compatible output)...${NC}"
-
-if ! command -v wasi-stub &> /dev/null; then
-    echo -e "${YELLOW}[WARN] wasi-stub not found. Installing...${NC}"
-    cargo install wasi-stub
-fi
-
-wasi-stub pkg/dot_agent_kernel_dsl_bg.wasm -o pkg/dot_agent_kernel_dsl_bg.wasm
-
-if [ $? -ne 0 ]; then
-    echo -e "${RED}[ERROR] wasi-stub failed.${NC}"
-    exit 1
-fi
-
 echo -e "${GREEN}[SUCCESS] Build complete! Artifacts in kernel-dsl/pkg/${NC}"
```

### 6.2 `index.js` and `index.browser.js` ŌĆö JavaScript WASI shim

A `wasiShim` object is added at module scope. It covers all 10 WASI functions needed by the `wasm32-wasip1` binary. Functions that write to WASM linear memory (`random_get`, `clock_time_get`, `environ_sizes_get`) use a lazy `_wasmExports` reference that is set immediately after `WebAssembly.instantiate` returns ŌĆö before any user-callable kernel API that would trigger WASI calls (HashMaps are created inside `AgentDSLKernel::new()`, not during WASM module initialization).

```js
let _wasmExports = null;

const wasiShim = {
  // HashMap seeding ŌĆö needs crypto-quality random bytes
  random_get(bufPtr, bufLen) {
    if (_wasmExports) crypto.getRandomValues(new Uint8Array(_wasmExports.memory.buffer, bufPtr, bufLen));
    return 0;
  },
  // Monotonic clock ŌĆö used by some internal Rust timing code
  clock_time_get(_id, _precision, timePtr) {
    if (_wasmExports) new DataView(_wasmExports.memory.buffer).setBigUint64(timePtr, BigInt(Date.now()) * 1_000_000n, true);
    return 0;
  },
  // No environment variables in browser/WASM context
  environ_get: () => 0,
  environ_sizes_get(countPtr, bufSizePtr) {
    if (_wasmExports) {
      const v = new DataView(_wasmExports.memory.buffer);
      v.setUint32(countPtr, 0, true);
      v.setUint32(bufSizePtr, 0, true);
    }
    return 0;
  },
  // No real filesystem ŌĆö return BADF (8) for all fd operations
  fd_close: () => 8,
  fd_prestat_get: () => 8,
  fd_prestat_dir_name: () => 8,
  fd_seek: () => 8,
  fd_write: () => 8,
  // Process exit ŌĆö throw JS exception to unwind the WASM call stack
  proc_exit: (code) => { throw new Error(`WASM exit: ${code}`); },
};
```

The `importObject` is updated to include `wasi_snapshot_preview1: wasiShim`, and `_wasmExports` is set right after instantiation:

```js
const wasmModule = await WebAssembly.instantiate(wasmBuffer, importObject);
_wasmExports = wasmModule.instance.exports;   // set BEFORE any kernel call
bgModule.__wbg_set_wasm(_wasmExports);
```

Memory growth safety: `_wasmExports.memory.buffer` returns the current `ArrayBuffer` on each access, so stale views after `memory.grow()` are not possible (consistent with the `patch-wasm-bindgen.js` fix already applied to `_bg.js`).

## 9. Upstream

The bug lives in `wasi-stub 0.3.0` (current latest). The WAT `[hash]` square-bracket syntax is a `wasm-tools` component model extension; wasi-stub's identifier parser does not handle it. Worth opening an issue at [astrale-sharp/wasm-minimal-protocol](https://github.com/astrale-sharp/wasm-minimal-protocol/tree/master/crates/wasi-stub) or its active fork [typst-community/wasm-minimal-protocol](https://github.com/typst-community/wasm-minimal-protocol).

If `wasi-stub` is fixed in a future release, the `build-wasm.sh` and `index.js`/`index.browser.js` could be reverted to the pre-shim form. The `--if-present` fix in `package.json` is independent and should remain.
